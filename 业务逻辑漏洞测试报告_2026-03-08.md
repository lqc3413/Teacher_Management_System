## 3. 详细问题

### B01 严重：登录角色参数可伪造，存在越权获取管理员权限

- 位置
  - `src/main/java/com/teacher/management/controller/UserController.java:70`
  - `src/main/java/com/teacher/management/controller/UserController.java:99`
  - `src/main/java/com/teacher/management/filter/JwtAuthenticationFilter.java:52`
- 问题描述
  - 登录逻辑只校验 `role == "admin"` 或 `role == "teacher"` 两种小写值。
  - 当传入 `role="ADMIN"`、`role="dept_director"` 等其他值时，未做角色合法性校验，直接写入 JWT。
  - JWT 过滤器会将 `role` 转为大写并赋权 `ROLE_...`，导致普通教师可伪造高权限角色。
- 复现思路
  1. 使用教师账号调用 `/api/users/login`，请求体传 `role: "ADMIN"`。
  2. 拿到 token 后访问 `/api/admin/**` 接口，可能被放行。
- 影响
  - 直接权限提升，属于高危越权漏洞。
- 修复建议
  - 忽略前端传入 `role`，统一以后端 `user.roleId` 映射角色并签发 token。
  - 对角色值做白名单校验，非法值直接拒绝。

### B02 严重：多个核心管理接口仅要求“已登录”，普通教师可执行管理操作

- 位置
  - `src/main/java/com/teacher/management/config/SecurityConfig.java:50`
  - `src/main/java/com/teacher/management/controller/UserController.java:142`
  - `src/main/java/com/teacher/management/controller/UserController.java:182`
  - `src/main/java/com/teacher/management/controller/UserController.java:194`
  - `src/main/java/com/teacher/management/controller/DepartmentController.java:74`
  - `src/main/java/com/teacher/management/controller/DepartmentController.java:80`
  - `src/main/java/com/teacher/management/controller/DepartmentController.java:86`
  - `src/main/java/com/teacher/management/controller/NoticeController.java:57`
  - `src/main/java/com/teacher/management/controller/NoticeController.java:69`
  - `src/main/java/com/teacher/management/controller/NoticeController.java:84`
  - `src/main/java/com/teacher/management/controller/SysConfigController.java:37`
- 问题描述
  - 安全配置仅将 `/api/admin/**`、`/api/dept-director/**`、`/api/teacher/**` 做了角色限制。
  - 其余 `/api/**` 一律只要求登录，导致普通教师也能调用用户管理、院系管理、公告管理、系统配置等写操作。
- 复现思路
  1. 教师账号登录。
  2. 使用教师 token 调用 `POST /api/users` 或 `PUT /api/config/batch` 等接口。
- 影响
  - 可篡改核心主数据、创建高权限账号、破坏系统配置，风险极高。
- 修复建议
  - 对上述接口统一加 `ADMIN` 鉴权（路由级或方法级 `@PreAuthorize`）。
  - 补充接口权限回归测试，避免新增接口漏配。

### B03 高：登录时未传 `role` 仍发放 token，后续请求会因空角色失效

- 位置
  - `src/main/java/com/teacher/management/controller/UserController.java:42`
  - `src/main/java/com/teacher/management/controller/UserController.java:99`
  - `src/main/java/com/teacher/management/filter/JwtAuthenticationFilter.java:48`
  - `src/main/java/com/teacher/management/filter/JwtAuthenticationFilter.java:52`
- 问题描述
  - `role` 可为空，登录仍会成功并签发 token。
  - 过滤器在处理 token 时对 `role.toUpperCase()`，`role` 为空会抛异常并清空认证上下文。
- 影响
  - 用户“登录成功但接口全不可用”，造成实际业务中断。
- 修复建议
  - 登录接口强制校验 `role` 非空且合法。
  - 或者完全不接收前端 `role`，由后端按账号角色签发。

### B04 高：提交详情对论文使用 `selectOne`，多论文场景会报错/丢数据

- 位置
  - `src/main/java/com/teacher/management/dto/TeacherSubmitDTO.java:19`
  - `src/main/java/com/teacher/management/service/TeacherService.java:260`
  - `src/main/java/com/teacher/management/service/TeacherService.java:412`
  - `src/main/java/com/teacher/management/service/TeacherService.java:498`
  - `src/main/java/com/teacher/management/vo/SubmissionDetailVO.java:32`
- 问题描述
  - 提交接口支持 `paperList`（多论文）并循环入库。
  - 详情接口却使用 `selectOne` 读取论文且 VO 只定义单个 `paper` 字段。
  - 当同一次提交包含多论文时，详情/导出可能抛出 `TooManyResultsException` 或仅返回一条。
- 影响
  - 审核、详情查看、导出结果不完整或直接失败。
- 修复建议
  - 详情 VO 改为 `List<PaperRecord> papers`。
  - 详情查询统一改 `selectList`，导出逻辑同步处理列表。

### B05 中：成果分组遗漏状态 `3/4`，导致业务数据展示不完整

- 位置
  - `src/main/java/com/teacher/management/entity/Submission.java:20`
  - `src/main/java/com/teacher/management/entity/Submission.java:22`
  - `src/main/java/com/teacher/management/service/TeacherService.java:529`
  - `src/main/java/com/teacher/management/service/TeacherService.java:541`
- 问题描述
  - 业务存在 `status=3(部门已通过待终审)`、`status=4(终审退回)`。
  - 成果分组仅返回 `0/1/2` 三类，`3/4` 记录没有被展示到任何分组。
- 影响
  - 教师端对审核进度认知错误，数据可见性不完整。
- 修复建议
  - 明确新增分组（如 `deptApproved`、`finalRejected`）或并入 `pending/rejected` 统一展示。

### B06 中：主任审核接口 `approved` 为空时触发空指针

- 位置
  - `src/main/java/com/teacher/management/controller/DeptDirectorController.java:207`
- 问题描述
  - 代码直接使用 `request.getApproved() ? 1 : 2`。
  - 若请求体未传 `approved` 或传 `null`，会触发 NPE。
- 影响
  - 审核接口返回 500，影响审核流程稳定性。
- 修复建议
  - 入参校验 `approved != null`，否则返回 4xx 业务错误。

### B07 中：管理员看板“待审核”统计口径错误（漏算 `status=3`）

- 位置
  - `src/main/java/com/teacher/management/controller/AdminController.java:280`
  - `src/main/java/com/teacher/management/controller/AdminController.java:197`
- 问题描述
  - 管理员终审可处理 `status=0` 和 `status=3`。
  - 但看板 `pendingCount` 仅统计了 `status=0`，导致待办数偏低。
- 影响
  - 管理驾驶舱指标失真，影响排班与审核节奏判断。
- 修复建议
  - 统计口径调整为 `status IN (0,3)`。

### B08 中（待确认规则）：被退回后可绕过任务时间窗口再次提交

- 位置
  - `src/main/java/com/teacher/management/service/TeacherService.java:100`
  - `src/main/java/com/teacher/management/service/TeacherService.java:106`
- 问题描述
  - 仅在“非退回重提”时校验任务状态与截止时间。
  - 退回重提路径可跳过时间校验，任务结束后仍可继续提交。
- 影响
  - 若业务要求“所有提交必须在任务窗口内”，则此为规则绕过。
- 修复建议
  - 与业务方确认规则；若不允许超时重提，则退回重提也要校验时间窗口。

## 4. 优先修复建议（按紧急度）

1. 先修 B01、B02（权限越权类，阻断高危风险）。
2. 再修 B04（会导致详情/审核/导出失败）。
3. 同步修 B03、B06（稳定性问题）。
4. 最后修 B05、B07、B08（口径与规则一致性）。
